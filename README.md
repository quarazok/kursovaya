# Быстрая доставка — CRM курьерской службы

Учебный проект курьерской службы: серверная часть на ASP.NET Web API и два веб-интерфейса — внутренняя CRM для сотрудников и публичный сайт для клиентов.

## Что умеет система

Сотрудники работают в CRM: ведут базу клиентов и курьеров, оформляют заказы, назначают на них доставки и проводят платежи. У администратора есть дополнительные права — он управляет учётными записями пользователей и видит общую статистику по системе.

Клиенты пользуются отдельным сайтом. Там они могут зарегистрироваться, оформить заказ (указав откуда и куда нужно доставить), отследить его статус и оплатить онлайн — наличными при получении, картой или переводом.

## Что под капотом

Бэкенд написан на ASP.NET Web API (.NET 10) с Entity Framework Core. Данные хранятся в MS SQL Server Express. Авторизация — JWT с хешированием паролей через BCrypt.

Фронтенд написан без фреймворков — обычный HTML, CSS и ванильный JavaScript. Это сделано намеренно, чтобы код был прозрачным и не требовал сборки.

## Запуск

### Подготовка базы данных

В SQL Server Management Studio подключись к экземпляру `.\SQLEXPRESS`. Саму базу создавать вручную не нужно — она появится при первой миграции.

Если хочешь использовать другой сервер, поправь строку подключения в `DeliveryApi/appsettings.json`:

```json
"ConnectionStrings": {
  "DefaultConnection": "Server=.\\SQLEXPRESS;Database=DeliveryDb;Trusted_Connection=True;TrustServerCertificate=True;"
}
```

### Применение миграций

```powershell
cd DeliveryApi
dotnet ef database update
```

Если команда `dotnet ef` не найдена, установи инструмент глобально:

```powershell
dotnet tool install --global dotnet-ef
```

### Запуск API

```powershell
cd DeliveryApi
dotnet run
```

Сервер поднимется на `http://localhost:5206`. По адресу `http://localhost:5206/swagger` доступен Swagger UI — там можно посмотреть все эндпоинты и потыкать их вручную.

### Запуск фронтенда

Открой папку `Frontend/` в VS Code и запусти Live Server (расширение «Live Server» от Ritwick Dey). Правой кнопкой по `Frontend/index.html` → «Open with Live Server». По умолчанию сайт открывается на `http://127.0.0.1:5500`.

Можно открывать HTML-файлы и двойным кликом, но через Live Server корректнее — не будет проблем с относительными путями.

## Роли пользователей

В системе три роли:

- **Client** — клиент. Создаёт и отслеживает свои заказы, оплачивает их.
- **Employee** — сотрудник. Работает в CRM: ведёт базу, назначает доставки, оформляет платежи.
- **Admin** — администратор. Всё, что может сотрудник, плюс управление учётками и просмотр статистики.

### Как сделать первого администратора

При обычной регистрации все сотрудники получают роль Employee. Чтобы превратить себя в админа, нужно вручную поменять роль в базе:

1. Зарегистрируйся как сотрудник на `http://127.0.0.1:5500/register.html`.
2. Подключись к базе `DeliveryDb` через SSMS.
3. Выполни запрос:

   ```sql
   UPDATE Users SET Role = 1 WHERE Email = 'твой@email.com';
   ```

   Значения роли в базе: `0` — Employee, `1` — Admin, `2` — Client.
4. Выйди из CRM и зайди заново. В шапке появится пункт «Пользователи».

## Как всё работает на практике

Сценарий обычного заказа выглядит так. Клиент регистрируется на своём сайте и оформляет заказ — указывает адрес отправки, адрес доставки и описание груза. Сотрудник в CRM видит новый заказ в общем списке. Он заходит на вкладку «Доставки», создаёт там новую доставку для этого заказа и назначает на неё свободного курьера.

Когда курьер забирает груз, сотрудник переводит доставку в статус «В пути». Клиент в это время видит у себя в кабинете, что курьер уже в пути. После доставки сотрудник отмечает доставку как завершённую, а потом выставляет счёт — создаёт платёж со статусом «Ожидает оплаты».

В личном кабинете клиента появляется кнопка «Оплатить». Клиент выбирает способ оплаты — наличные, карта или перевод — и подтверждает. Платёж переходит в статус «Оплачен», а сотрудники видят оплаченный заказ в CRM.

## API

Эндпоинты разделены по областям. Все вызовы, кроме регистрации и логина, требуют JWT-токен в заголовке `Authorization: Bearer <token>`.

### Авторизация

| Метод | Путь                  | Доступ    | Описание                  |
|-------|-----------------------|-----------|---------------------------|
| POST  | `/api/auth/register`  | без авт.  | регистрация сотрудника    |
| POST  | `/api/auth/login`     | без авт.  | вход (любая роль)         |

### CRM (роль Employee или Admin)

Для каждой сущности доступен полный CRUD: получение списка, получение по id, создание, изменение и удаление.

| Сущность   | Базовый путь        |
|------------|---------------------|
| Клиенты    | `/api/clients`      |
| Курьеры    | `/api/couriers`     |
| Заказы     | `/api/orders`       |
| Доставки   | `/api/deliveries`   |
| Платежи    | `/api/payments`     |

### Управление пользователями (только Admin)

| Метод  | Путь                       | Описание              |
|--------|----------------------------|-----------------------|
| GET    | `/api/users`               | список пользователей  |
| GET    | `/api/users/stats`         | статистика системы    |
| PATCH  | `/api/users/{id}/role`     | сменить роль          |
| DELETE | `/api/users/{id}`          | удалить пользователя  |

### Клиентский портал (роль Client)

| Метод | Путь                                  | Описание           |
|-------|---------------------------------------|--------------------|
| POST  | `/api/clientportal/register`          | регистрация        |
| GET   | `/api/clientportal/profile`           | профиль            |
| GET   | `/api/clientportal/orders`            | мои заказы         |
| POST  | `/api/clientportal/orders`            | создать заказ      |
| GET   | `/api/clientportal/orders/{id}`       | отследить заказ    |
| POST  | `/api/clientportal/orders/{id}/pay`   | оплатить заказ     |

## Модель данных

Связи между таблицами устроены так:

- **User** — учётная запись с email, паролем и ролью.
- **Client** — карточка клиента (имя, телефон, email). У клиентов, которые зарегистрировались сами, есть внешний ключ `UserId` на таблицу пользователей.
- **Courier** — курьер.
- **Order** — заказ клиента: адреса, описание, статус.
- **Delivery** — назначение курьера на конкретный заказ. У одной доставки один курьер и один заказ.
- **Payment** — счёт по заказу: сумма, способ оплаты, статус.

Один клиент может иметь много заказов. У одного заказа — одна доставка и один платёж.

## Конфигурация

Основные параметры лежат в `DeliveryApi/appsettings.json`:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.\\SQLEXPRESS;Database=DeliveryDb;Trusted_Connection=True;TrustServerCertificate=True;"
  },
  "Jwt": {
    "Key": "DeliveryApp_SuperSecretKey_2024_ForStudentProject!",
    "Issuer": "DeliveryApi",
    "Audience": "DeliveryFrontend"
  }
}
```

CORS настроен на разрешение любых источников. Для учебного проекта это нормально, но в продакшене стоит ограничить список доменов.

## Если что-то не работает

**API не запускается, пишет, что файл занят.** Это значит, что предыдущий процесс ещё не завершился. Останови его:

```powershell
Stop-Process -Name "DeliveryApi" -Force
dotnet run
```

**Фронт не достучится до API.** Проверь, что сервер действительно запущен — открой `http://localhost:5206/swagger`. Если страница открылась, дело не в сервере. Если нет — запусти заново через `dotnet run`.

**Сразу кидает на страницу входа.** Если ты на CRM-странице (`dashboard.html`, `clients.html` и т.п.) и не залогинен — это правильное поведение, надо сначала войти. Если ты на клиентском сайте (`/client/index.html`), редиректа быть не должно — проверь URL в адресной строке.

**Live Server показывает не то.** Открывай в VS Code именно папку `Frontend/`, а не корень проекта. Тогда корнем сайта будет нужная папка и относительные пути сработают.
