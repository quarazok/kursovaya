Основное
Архитектура веб-приложения: большая картина
Зачем делить приложение на части?
Представь, что ты пишешь всё в одном месте: и работу с базой данных, и логику приложения, и то, что видит пользователь. Поначалу это кажется удобным. Но когда приложение начинает расти, такой код превращается в хаос: изменение одной строчки ломает что-то совершенно в другом месте, а понять, кто за что отвечает, становится невозможно.
Поэтому профессиональные приложения разделяют на три независимых слоя, каждый из которых отвечает строго за своё:
Слой
	Технология
	Роль
	База данных
	MS SQL
	Хранит данные
	Backend / API
	ASP.NET
	Обрабатывает логику, отвечает на запросы
	Frontend
	React, Vue, Angular и др.
	Показывает интерфейс пользователю
	Это разделение называется трёхслойной архитектурой (three-tier architecture). Она является стандартом в индустрии — большинство современных веб-приложений устроены именно так.


________________
Как это работает: путь одного действия
Разберём конкретный пример. Пользователь открывает страницу интернет-магазина и нажимает кнопку "Показать список товаров".
Вот что происходит шаг за шагом.
Шаг 1. Пользователь — Frontend
Пользователь видит страницу в браузере. Это и есть Frontend — то, что нарисовано на экране. Когда пользователь нажимает кнопку, Frontend не идёт в базу данных напрямую. Он не знает, где она находится и как с ней говорить. Вместо этого он отправляет HTTP-запрос к API.
HTTP-запрос выглядит примерно так:
GET https://myapp.com/api/products
Это как письмо: "Привет, API. Дай мне список товаров, пожалуйста."
Шаг 2. Frontend — Backend (ASP.NET API)
API получает запрос. Теперь его задача — понять, что нужно сделать, проверить права доступа, применить бизнес-логику и сходить в базу данных за данными.
Например, в коде это выглядит как контроллер:
[HttpGet("/api/products")]
public IActionResult GetProducts()
{
    var products = _productService.GetAll();
    return Ok(products);
}
API не возвращает HTML-страницу. Он возвращает данные в формате JSON — универсальном текстовом формате, который понимают все языки и все фронтенды.
________________
Шаг 3. Backend — База данных (MS SQL)
Чтобы получить товары, API отправляет SQL-запрос к базе данных:
SELECT id, name, price FROM Products;
База данных выполняет запрос и возвращает строки таблицы обратно в API.
Шаг 4. Обратный путь
Данные идут в обратном направлении:
1. База данных возвращает строки -> API
2. API упаковывает их в JSON и возвращает -> Frontend
3. Frontend получает JSON, разбирает его и рисует список товаров на экране
JSON, который получает Frontend, выглядит так:
[
  { "id": 1, "name": "Ноутбук", "price": 75000 },
  { "id": 2, "name": "Мышь", "price": 1200 }
]
Frontend берёт эти данные и превращает их в карточки, таблицу или список — в зависимости от того, как написан интерфейс.


________________
Почему каждый слой не лезет в чужую работу?
Это ключевое правило архитектуры. Каждый слой общается только с соседним — и никогда не перепрыгивает через него.
Frontend никогда не ходит в базу данных напрямую. Это было бы небезопасно: любой пользователь мог бы видеть настоящие SQL-запросы и данные, которые ему видеть не положено.
База данных не знает, кто к ней обращается. Ей всё равно, это API или что-то другое. Она просто выполняет запросы.
API — единственный, кто знает всё. Он стоит посередине, проверяет права, применяет правила и решает, какие данные отдать и в каком виде.
Итог
Когда ты смотришь на любое современное веб-приложение, за ним стоит эта же схема. Неважно, Twitter это или корпоративная CRM-система. Данные всегда идут по одному пути:
Пользователь -> Frontend -> API -> База данных -> API -> Frontend -> Пользователь




База данных
Блок 2. База данных — MS SQL
Что такое база данных и зачем она нужна
База данных — это организованное хранилище данных. Не просто файл с текстом, а система, которая умеет быстро искать, фильтровать, сортировать и связывать данные между собой.
Без базы данных приложение не может "помнить" ничего между запусками. Каждый раз при перезапуске сервера все данные терялись бы. База данных решает именно эту проблему: она хранит данные независимо от того, работает приложение или нет.
MS SQL (Microsoft SQL Server) — одна из самых популярных реляционных баз данных. Она используется в корпоративных приложениях, банках, государственных системах. Язык, на котором с ней общаются — SQL (Structured Query Language).
Как устроены данные внутри
Таблица — основная единица хранения
Все данные в реляционной базе хранятся в таблицах. Таблица — это как лист Excel: есть столбцы (колонки) и строки.
Пример таблицы Products:
id
	name
	price
	stock
	1
	Ноутбук
	75000
	10
	2
	Мышь
	1200
	150
	3
	Клавиатура
	3500
	80
	Каждый столбец — это отдельный тип данных (число, текст, дата и т.д.). Каждая строка — это одна запись (один конкретный товар).
Типы данных
При создании таблицы нужно явно указать, какой тип данных хранится в каждом столбце. MS SQL не разрешает положить текст в числовой столбец.
Самые распространённые типы:

Первичный ключ (Primary Key)
Каждая строка в таблице должна быть уникально идентифицирована. Для этого существует первичный ключ (PK) — столбец, значение которого уникально для каждой строки и никогда не повторяется.
Обычно это столбец id с типом INT и автоматическим инкрементом — то есть база данных сама присваивает новый номер каждой добавленной строке.
CREATE TABLE Products (
    id    INT           PRIMARY KEY IDENTITY(1,1),
    name  NVARCHAR(255) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock INT           NOT NULL DEFAULT 0
);
Что здесь написано построчно:
* id INT PRIMARY KEY IDENTITY(1,1) — столбец id, тип целое число, первичный ключ, автоматически увеличивается на 1 при каждой новой записи (начиная с 1)
* name NVARCHAR(255) NOT NULL — текстовый столбец, максимум 255 символов, не может быть пустым
* price DECIMAL(10,2) NOT NULL — число с двумя знаками после запятой, обязательное
* stock INT NOT NULL DEFAULT 0 — целое число, по умолчанию 0, если значение не передано
________________
Внешний ключ (Foreign Key) и связи между таблицами
Реляционная база данных сильна тем, что таблицы могут быть связаны между собой. Это позволяет не дублировать данные.
Допустим, у нас есть таблица заказов Orders. Каждый заказ принадлежит конкретному пользователю. Вместо того чтобы хранить имя и email пользователя в каждой строке заказа, мы просто храним его id — ссылку на строку в таблице Users.
Таблица Users:
id
	name
	email
	1
	Анна
	anna@mail.com
	2
	Игорь
	igor@mail.com
	Таблица Orders:
id
	user_id
	total
	created_at
	1
	1
	76200
	2024-03-10 12:00:00
	2
	1
	3500
	2024-03-12 15:30:00
	3
	2
	1200
	2024-03-14 09:00:00
	Столбец user_id в таблице Orders — это внешний ключ (FK). Он ссылается на id в таблице Users. База данных следит за тем, чтобы в user_id нельзя было записать id несуществующего пользователя.
CREATE TABLE Orders (
    id         INT           PRIMARY KEY IDENTITY(1,1),
    user_id    INT           NOT NULL,
    total      DECIMAL(10,2) NOT NULL,
    created_at DATETIME      NOT NULL DEFAULT GETDATE(),


    CONSTRAINT fk_orders_users
        FOREIGN KEY (user_id) REFERENCES Users(id)
);
Что здесь написано построчно:
* user_id INT NOT NULL — столбец для хранения ссылки на пользователя
* DEFAULT GETDATE() — если дата не передана, база сама подставит текущее время
* CONSTRAINT fk_orders_users — даём имя ограничению (удобно при отладке)
* FOREIGN KEY (user_id) REFERENCES Users(id) — говорим базе: значение в user_id обязано существовать в столбце id таблицы Users

Основные SQL-запросы
SELECT — получить данные
-- Все товары
SELECT * FROM Products;
-- Только название и цену
SELECT name, price FROM Products;


-- Товары дороже 5000
SELECT name, price FROM Products
WHERE price > 5000;


-- Отсортировать по цене по убыванию
SELECT name, price FROM Products
ORDER BY price DESC;


-- Первые 10 записей
SELECT TOP 10 name, price FROM Products
ORDER BY price DESC;
INSERT — добавить запись
INSERT INTO Products (name, price, stock)
VALUES ('Монитор', 25000, 30);
Обрати внимание: id не указываем — база присвоит его сама благодаря IDENTITY.
UPDATE — изменить запись
UPDATE Products
SET price = 24000, stock = 25
WHERE id = 4;
Никогда не пиши UPDATE без WHERE — иначе изменятся все строки таблицы сразу.
DELETE — удалить запись
DELETE FROM Products
WHERE id = 4;
То же правило: DELETE без WHERE удалит все строки.
JOIN — объединить данные из двух таблиц
Самый важный и часто используемый приём. Позволяет получить данные сразу из нескольких связанных таблиц.
-- Получить все заказы вместе с именем пользователя
SELECT
    Orders.id        AS order_id,
    Users.name       AS user_name,
    Orders.total,
    Orders.created_at
FROM Orders
INNER JOIN Users ON Orders.user_id = Users.id;
________________


Результат:
order_id
	user_name
	total
	created_at
	1
	Анна
	76200
	2024-03-10 12:00:00
	2
	Анна
	3500
	2024-03-12 15:30:00
	3
	Игорь
	1200
	2024-03-14 09:00:00
	INNER JOIN возвращает только те строки, у которых есть совпадение в обеих таблицах. Если у пользователя нет заказов — он не попадёт в результат.
Существуют и другие виды JOIN:
* LEFT JOIN — вернёт всех пользователей, даже если у них нет заказов (поле заказа будет NULL)
* RIGHT JOIN — вернёт все заказы, даже если пользователь не найден
* FULL JOIN — вернёт всё из обеих таблиц
Агрегатные функции
-- Сколько всего товаров
SELECT COUNT(*) FROM Products;


-- Средняя цена
SELECT AVG(price) FROM Products;


-- Сумма всех заказов по каждому пользователю
SELECT
    Users.name,
    SUM(Orders.total) AS total_spent
FROM Orders
INNER JOIN Users ON Orders.user_id = Users.id
GROUP BY Users.name;
Индексы
Когда в таблице миллион строк, запрос WHERE name = 'Ноутбук' будет проверять каждую строку по одной. Это медленно.
Индекс — это дополнительная структура данных, которую база строит для быстрого поиска. Представь алфавитный указатель в конце книги: вместо перелистывания всей книги ты сразу находишь нужную страницу.
-- Создать индекс по полю name в таблице Products
CREATE INDEX idx_products_name ON Products(name);
Первичный ключ (PRIMARY KEY) всегда индексируется автоматически. Внешние ключи рекомендуется индексировать вручную — это ускорит JOIN-запросы.
Индексы ускоряют чтение, но немного замедляют запись (база должна обновлять индекс при каждом INSERT и UPDATE). Поэтому не нужно создавать индекс на каждый столбец — только на те, по которым часто ищут.


Транзакции
Иногда нужно выполнить несколько запросов как одно неделимое действие. Классический пример: перевод денег между счетами. Нужно списать с одного счёта И зачислить на другой. Если второй запрос упадёт с ошибкой, первый тоже должен откатиться.
Для этого существуют транзакции.
BEGIN TRANSACTION;


UPDATE Accounts SET balance = balance - 5000 WHERE id = 1;
UPDATE Accounts SET balance = balance + 5000 WHERE id = 2;


COMMIT; -- Если всё прошло успешно — фиксируем изменения
________________


Если что-то пошло не так:
BEGIN TRANSACTION;


UPDATE Accounts SET balance = balance - 5000 WHERE id = 1;
-- Здесь возникла ошибка


ROLLBACK; -- Отменяем все изменения внутри транзакции
Правило: если все шаги выполнены успешно — COMMIT. Если хоть один упал — ROLLBACK.
________________


Типичные ошибки новичков
1. UPDATE и DELETE без WHERE
-- ОПАСНО: обновит ВСЕ строки таблицы
UPDATE Products SET price = 0;


-- ПРАВИЛЬНО: обновляет только конкретную запись
UPDATE Products SET price = 0 WHERE id = 5;
2. Хранение связанных данных в одном столбце через запятую
-- ПЛОХО: список id товаров в одном столбце
Orders: id=1, product_ids="1,2,3"


-- ПРАВИЛЬНО: отдельная таблица связей
CREATE TABLE OrderItems (
    order_id   INT NOT NULL,
    product_id INT NOT NULL,
    quantity   INT NOT NULL
);
Когда данные хранятся через запятую, их невозможно нормально искать, фильтровать и связывать. Всегда создавай отдельную таблицу для связи многие-ко-многим.
3. NULL не равен NULL
В SQL NULL означает "значение отсутствует". Сравнивать с ним нужно не через =, а через IS NULL.
-- Не работает как ожидается
SELECT * FROM Products WHERE description = NULL;


-- Правильно
SELECT * FROM Products WHERE description IS NULL;
4. Отсутствие индексов на внешних ключах
-- После создания внешнего ключа — добавь индекс
CREATE INDEX idx_orders_user_id ON Orders(user_id);
Без индекса каждый JOIN будет перебирать всю таблицу.
5. Хранение паролей в открытом виде
Никогда не записывай пароль пользователя напрямую в базу данных. Всегда храни только хеш пароля, вычисленный через алгоритм bcrypt или аналогичный. Это задача Backend, но важно понимать на уровне архитектуры.
Итог
База данных — это фундамент приложения. Она не знает ни про фронтенд, ни про бизнес-логику. Её единственная задача — надёжно хранить данные и быстро их отдавать по запросу.
Ключевые идеи этого раздела:
* Данные хранятся в таблицах (строки и столбцы)
* Каждая строка идентифицируется первичным ключом
* Таблицы связываются через внешние ключи
* SQL — язык для работы с данными: SELECT, INSERT, UPDATE, DELETE, JOIN
* Индексы ускоряют поиск
* Транзакции гарантируют целостность данных при нескольких операциях


API
Блок 3. Backend — ASP.NET API
Что такое backend и зачем он нужен
Backend — это программа, которая работает на сервере. Пользователь её не видит. Она не рисует кнопки и не показывает страницы. Её задача — принять запрос, обработать его, сходить в базу данных и вернуть данные.
Если Frontend — это витрина магазина, то Backend — это склад и кассир за стеной. Покупатель (пользователь) не заходит на склад напрямую. Он говорит кассиру что хочет, кассир идёт на склад, берёт нужное и приносит обратно.
ASP.NET — это фреймворк от Microsoft для создания backend-приложений на языке C#. Когда говорят "ASP.NET API", имеют в виду backend-приложение, которое принимает HTTP-запросы и возвращает данные в формате JSON.
Часть 1. HTTP — как браузер разговаривает с сервером
Прежде чем писать код, нужно понять протокол, на котором всё построено.
Что такое HTTP
HTTP (HyperText Transfer Protocol) — это набор правил, по которым клиент (браузер или фронтенд) и сервер (backend) обмениваются сообщениями. Это как язык: обе стороны договорились, что запросы и ответы будут выглядеть определённым образом.
Каждое взаимодействие состоит из двух частей:
* Запрос (Request) — клиент говорит серверу что ему нужно
* Ответ (Response) — сервер отвечает
________________
Структура HTTP-запроса
Когда фронтенд отправляет запрос, он выглядит примерно так:
GET /api/products HTTP/1.1
Host: myapp.com
Content-Type: application/json
Authorization: Bearer eyJhbGci...
Разберём построчно:
* GET — метод запроса (что хотим сделать)
* /api/products — путь (к какому ресурсу обращаемся)
* HTTP/1.1 — версия протокола
* Host: myapp.com — адрес сервера
* Content-Type: application/json — говорим серверу, что отправляем JSON
* Authorization: Bearer ... — токен авторизации (если нужен)
HTTP-методы
Метод говорит серверу, что именно хочет сделать клиент. Их четыре основных:
Метод
	Что означает
	Пример
	GET
	Получить данные
	Показать список товаров
	POST
	Создать новую запись
	Добавить новый товар
	PUT
	Обновить существующую запись
	Изменить цену товара
	DELETE
	Удалить запись
	Удалить товар
	Это договорённость, которую принято соблюдать. Технически можно сделать удаление через GET — но это будет неправильно и запутает всех, кто будет читать твой код.
Структура HTTP-ответа
Когда сервер отвечает, ответ выглядит так:
HTTP/1.1 200 OK
Content-Type: application/json


[
  { "id": 1, "name": "Ноутбук", "price": 75000 },
  { "id": 2, "name": "Мышь", "price": 1200 }
]
* 200 OK — статус-код (всё прошло успешно)
* Content-Type: application/json — говорим клиенту, что возвращаем JSON
* Дальше идёт тело ответа — сами данные
________________
Статус-коды
Статус-код — это трёхзначное число, которое говорит клиенту, что произошло на сервере.
Код
	Значение
	Когда используется
	200 OK
	Успешно
	Данные получены
	201 Created
	Создано
	Новая запись добавлена
	204 No Content
	Успешно, нет данных
	Удаление прошло успешно
	400 Bad Request
	Плохой запрос
	Клиент прислал неверные данные
	401 Unauthorized
	Не авторизован
	Нет или неверный токен
	403 Forbidden
	Нет доступа
	Токен есть, но прав не хватает
	404 Not Found
	Не найдено
	Запись с таким id не существует
	500 Internal Server Error
	Ошибка сервера
	Что-то сломалось на backend
	Коды 2xx — всё хорошо. Коды 4xx — ошибка на стороне клиента. Коды 5xx — ошибка на стороне сервера.
Часть 2. Минимум C# для понимания кода
Если C# не знаком совсем, перед тем как читать код контроллеров, нужно понять несколько базовых вещей.
Классы и объекты
C# — объектно-ориентированный язык. Почти всё в нём — это классы.
Класс — это шаблон. Объект — это конкретный экземпляр этого шаблона.
// Это класс — шаблон для описания товара
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public int Stock { get; set; }
}
Разберём построчно:
* public class Product — объявляем класс с именем Product, слово public означает что он доступен из других файлов
* public int Id { get; set; } — свойство Id типа int (целое число). { get; set; } означает что значение можно и читать, и записывать
* public string Name { get; set; } — свойство Name типа string (текст)
* public decimal Price { get; set; } — свойство Price типа decimal (число с дробной частью, удобно для цен)
Когда мы получаем товар из базы данных, мы создаём объект этого класса:
Product product = new Product();
product.Id = 1;
product.Name = "Ноутбук";
product.Price = 75000;
Методы
Метод — это функция внутри класса. Она принимает аргументы и возвращает результат.
// Метод принимает число и возвращает число
public decimal ApplyDiscount(decimal price, decimal discountPercent)
{
    decimal discount = price * discountPercent / 100;
    return price - discount;
}
Разберём построчно:
* public — метод доступен снаружи класса
* decimal — тип возвращаемого значения (что метод вернёт)
* ApplyDiscount — имя метода
* (decimal price, decimal discountPercent) — аргументы: два числа типа decimal
* Внутри фигурных скобок {} — тело метода, сам код
* return — что именно вернуть
Если метод ничего не возвращает, вместо типа пишут void:
public void PrintName(string name)
{
    Console.WriteLine(name);
    // return здесь не нужен
}
Списки
Очень часто из базы данных приходит не один объект, а много. Для этого используют List<T>, где T — тип элементов.
// Список товаров
List<Product> products = new List<Product>();


// Добавить элемент
products.Add(new Product { Id = 1, Name = "Ноутбук", Price = 75000 });


// Получить первый элемент
Product first = products[0];


// Узнать количество
int count = products.Count;
async / await
В реальных приложениях запросы к базе данных занимают время. Пока идёт запрос, сервер не должен "зависать" и отказывать другим пользователям. Для этого используют асинхронный код.
// Синхронный метод — ждёт результата и блокирует поток
public List<Product> GetProducts()
{
    return _db.Products.ToList();
}


// Асинхронный метод — ждёт результата, но не блокирует поток
public async Task<List<Product>> GetProductsAsync()
{
    return await _db.Products.ToListAsync();
}
Разберём разницу:
* async перед методом — говорит C# что метод асинхронный
* Task<List<Product>> — возвращаемый тип. Task — это "обещание" что результат будет, просто не сейчас
* await перед операцией — "подожди здесь, пока не придёт результат, но пока ждёшь — занимайся другими запросами"
Правило простое: если метод использует await внутри, он должен быть помечен как async. Если метод async — его возвращаемый тип оборачивается в Task<>.

Часть 3. Структура ASP.NET проекта
Когда создаёшь новый проект ASP.NET Web API, Visual Studio генерирует набор файлов. Вот самые важные:
MyApp/
├── Program.cs              <- Точка входа, настройка приложения
├── appsettings.json        <- Конфигурация (строка подключения к БД и др.)
├── Controllers/
│   └── ProductsController.cs  <- Обработчики HTTP-запросов
├── Models/
│   └── Product.cs          <- Классы, описывающие данные
└── Data/
    └── AppDbContext.cs     <- Подключение к базе данных (Entity Framework)
Каждая папка отвечает за своё:
* Controllers — принимают запросы и возвращают ответы
* Models — описывают структуру данных (как выглядит товар, пользователь и т.д.)
* Data — всё, что связано с подключением к базе данных
Часть 4. Entity Framework — работа с базой данных без SQL
Что такое ORM
Обычно, чтобы получить данные из базы, нужно писать SQL-запросы вручную. Entity Framework (EF) — это ORM (Object-Relational Mapper), библиотека которая берёт эту работу на себя.
Вместо того чтобы писать:
SELECT * FROM Products WHERE price > 5000;
Ты пишешь на C#:
var products = await _db.Products
    .Where(p => p.Price > 5000)
    .ToListAsync();
EF сам переводит это в SQL, отправляет в базу и возвращает тебе список объектов Product. Тебе не нужно думать о синтаксисе SQL для простых запросов.
DbContext — главный класс для работы с БД
DbContext — это класс, который представляет подключение к базе данных. Через него EF знает, какие таблицы существуют и как они выглядят.
// Data/AppDbContext.cs
using Microsoft.EntityFrameworkCore;


public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }


    // Каждое свойство DbSet<T> — это одна таблица в базе данных
    public DbSet<Product> Products { get; set; }
    public DbSet<User> Users { get; set; }
    public DbSet<Order> Orders { get; set; }
}
Разберём построчно:
* public class AppDbContext : DbContext — наш класс наследуется от DbContext. Двоеточие : в C# означает "наследуется от"
* Конструктор AppDbContext(DbContextOptions<AppDbContext> options) : base(options) — принимает настройки подключения и передаёт их родительскому классу
* public DbSet<Product> Products { get; set; } — говорим EF: в базе есть таблица, и работать с ней мы будем через коллекцию объектов Product
Подключение к базе данных
Строка подключения хранится в файле appsettings.json:
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=MyAppDb;Trusted_Connection=True;"
  }
}
Разберём строку подключения:
* Server=localhost — адрес сервера с MS SQL (localhost — это твой собственный компьютер)
* Database=MyAppDb — имя базы данных
* Trusted_Connection=True — использовать Windows-аутентификацию (без пароля)
В файле Program.cs регистрируем DbContext:
// Program.cs
var builder = WebApplication.CreateBuilder(args);


// Регистрируем DbContext и говорим ему использовать MS SQL
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("DefaultConnection")
    )
);


builder.Services.AddControllers();


var app = builder.Build();


app.MapControllers();
app.Run();
Разберём построчно:
* WebApplication.CreateBuilder(args) — создаём строитель приложения
* builder.Services.AddDbContext<AppDbContext>(...) — регистрируем наш DbContext в системе. После этого ASP.NET сможет автоматически передавать его в контроллеры
* options.UseSqlServer(...) — говорим EF что будем использовать именно MS SQL
* builder.Configuration.GetConnectionString("DefaultConnection") — читаем строку подключения из appsettings.json
* builder.Services.AddControllers() — говорим приложению что у нас есть контроллеры
* app.MapControllers() — включаем маршрутизацию запросов к контроллерам
* app.Run() — запускаем приложение
Migrations — как EF создаёт таблицы
EF умеет сам создавать таблицы в базе данных на основе твоих классов-моделей. Это называется миграции.
Последовательность действий:
1. Ты описываешь модель в C# (класс Product)
2. Запускаешь команду Add-Migration — EF генерирует файл с инструкциями для создания таблицы
3. Запускаешь команду Update-Database — EF выполняет эти инструкции и таблица появляется в базе
# В Package Manager Console (Visual Studio)
Add-Migration InitialCreate
Update-Database
После этого в базе появятся таблицы Products, Users, Orders — точно такие, как описано в DbContext.
Часть 5. Контроллеры — сердце API
Что такое контроллер
Контроллер — это класс, который принимает HTTP-запросы и возвращает HTTP-ответы. Каждый метод внутри контроллера — это один endpoint (маршрут), то есть один адрес, по которому можно обратиться к API.
Пример: контроллер для работы с товарами.
// Controllers/ProductsController.cs
using Microsoft.AspNetCore.Mvc;
using Microsoft.EntityFrameworkCore;


[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly AppDbContext _db;


    public ProductsController(AppDbContext db)
    {
        _db = db;
    }


    // GET api/products
    [HttpGet]
    public async Task<IActionResult> GetAll()
    {
        var products = await _db.Products.ToListAsync();
        return Ok(products);
    }


    // GET api/products/5
    [HttpGet("{id}")]
    public async Task<IActionResult> GetById(int id)
    {
        var product = await _db.Products.FindAsync(id);


        if (product == null)
            return NotFound();


        return Ok(product);
    }


    // POST api/products
    [HttpPost]
    public async Task<IActionResult> Create([FromBody] Product product)
    {
        _db.Products.Add(product);
        await _db.SaveChangesAsync();
        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }


    // PUT api/products/5
    [HttpPut("{id}")]
    public async Task<IActionResult> Update(int id, [FromBody] Product updated)
    {
        var product = await _db.Products.FindAsync(id);


        if (product == null)
            return NotFound();


        product.Name  = updated.Name;
        product.Price = updated.Price;
        product.Stock = updated.Stock;


        await _db.SaveChangesAsync();
        return Ok(product);
    }


    // DELETE api/products/5
    [HttpDelete("{id}")]
    public async Task<IActionResult> Delete(int id)
    {
        var product = await _db.Products.FindAsync(id);


        if (product == null)
            return NotFound();


        _db.Products.Remove(product);
        await _db.SaveChangesAsync();
        return NoContent();
    }
}
Теперь разберём каждую часть этого кода подробно.
Атрибуты над классом
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
* [ApiController] — атрибут, который говорит ASP.NET: "этот класс — контроллер API". Он включает ряд удобных проверок автоматически
* [Route("api/[controller]")] — базовый маршрут для всех методов этого контроллера. [controller] автоматически заменяется на имя класса без слова Controller. То есть ProductsController → маршрут будет api/products
* : ControllerBase — контроллер наследуется от базового класса, который даёт методы Ok(), NotFound(), NoContent() и другие
Конструктор и внедрение зависимостей
private readonly AppDbContext _db;


public ProductsController(AppDbContext db)
{
    _db = db;
}
* private readonly AppDbContext _db — объявляем поле класса. private — доступно только внутри этого класса. readonly — нельзя переназначить после конструктора
* Конструктор принимает AppDbContext db — ASP.NET сам создаёт объект AppDbContext и передаёт его сюда. Это называется Dependency Injection (внедрение зависимостей). Тебе не нужно писать new AppDbContext() — фреймворк делает это за тебя
* _db = db — сохраняем переданный контекст в поле, чтобы использовать его в других методах
GET — получить все записи
[HttpGet]
public async Task<IActionResult> GetAll()
{
    var products = await _db.Products.ToListAsync();
    return Ok(products);
}
* [HttpGet] — этот метод отвечает на GET-запросы по адресу api/products
* async Task<IActionResult> — метод асинхронный, возвращает HTTP-ответ (IActionResult — интерфейс для любого типа ответа)
* _db.Products.ToListAsync() — просим EF выполнить запрос SELECT * FROM Products и вернуть список объектов
* return Ok(products) — возвращаем ответ со статусом 200 OK и телом в виде JSON-списка товаров
GET — получить одну запись по id
[HttpGet("{id}")]
public async Task<IActionResult> GetById(int id)
{
    var product = await _db.Products.FindAsync(id);


    if (product == null)
        return NotFound();


    return Ok(product);
}
* [HttpGet("{id}")] — маршрут api/products/5. Фигурные скобки {id} означают что это переменная часть URL
* int id — ASP.NET автоматически берёт значение 5 из URL и передаёт его в параметр id
* FindAsync(id) — ищет запись с таким id. Если не найдена — возвращает null
* if (product == null) return NotFound() — возвращаем 404 Not Found если товар не существует
* return Ok(product) — возвращаем 200 OK с данными товара
POST — создать новую запись
[HttpPost]
public async Task<IActionResult> Create([FromBody] Product product)
{
    _db.Products.Add(product);
    await _db.SaveChangesAsync();
    return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
}
* [HttpPost] — этот метод отвечает на POST-запросы по адресу api/products
* [FromBody] Product product — говорим ASP.NET взять тело запроса (JSON) и автоматически превратить его в объект Product. [FromBody] означает "читай из тела запроса"
* _db.Products.Add(product) — добавляем товар в контекст EF (пока только в памяти, не в базе)
* await _db.SaveChangesAsync() — теперь EF выполняет реальный INSERT INTO Products ... в базу данных
* return CreatedAtAction(...) — возвращаем 201 Created. Первый аргумент — имя метода для получения созданного объекта, второй — его id, третий — сам объект
PUT — обновить существующую запись
[HttpPut("{id}")]
public async Task<IActionResult> Update(int id, [FromBody] Product updated)
{
    var product = await _db.Products.FindAsync(id);


    if (product == null)
        return NotFound();


    product.Name  = updated.Name;
    product.Price = updated.Price;
    product.Stock = updated.Stock;


    await _db.SaveChangesAsync();
    return Ok(product);
}
* Сначала находим существующий товар по id
* Если не найден — возвращаем 404
* Обновляем поля найденного объекта значениями из запроса
* SaveChangesAsync() — EF видит что объект изменился и выполняет UPDATE Products SET ... WHERE id = ...
* Возвращаем обновлённый объект
DELETE — удалить запись
[HttpDelete("{id}")]
public async Task<IActionResult> Delete(int id)
{
    var product = await _db.Products.FindAsync(id);


    if (product == null)
        return NotFound();


    _db.Products.Remove(product);
    await _db.SaveChangesAsync();
    return NoContent();
}
* Находим товар, если нет — 404
* _db.Products.Remove(product) — помечаем объект для удаления
* SaveChangesAsync() — EF выполняет DELETE FROM Products WHERE id = ...
* return NoContent() — возвращаем 204 No Content: успешно, но возвращать нечего
Часть 6. Модели и DTO
Зачем нужны DTO
Product — это модель базы данных. Она напрямую отражает таблицу. Но часто возникает ситуация когда клиенту не нужны все поля, или наоборот — нужно вернуть поля из нескольких таблиц сразу.
Для этого используют DTO (Data Transfer Object) — отдельный класс специально для передачи данных клиенту.
Пример: при возврате списка товаров не нужно отдавать количество на складе — это внутренняя информация.
// Модель базы данных — полная
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    public int Stock { get; set; }          // внутренняя информация
    public DateTime CreatedAt { get; set; } // внутренняя информация
}


// DTO — только то, что нужно клиенту
public class ProductDto
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
}
В контроллере преобразуем модель в DTO перед отправкой:
[HttpGet]
public async Task<IActionResult> GetAll()
{
    var products = await _db.Products.ToListAsync();


    // Преобразуем каждый Product в ProductDto
    var result = products.Select(p => new ProductDto
    {
        Id    = p.Id,
        Name  = p.Name,
        Price = p.Price
    }).ToList();


    return Ok(result);
}
* products.Select(...) — метод который трансформирует каждый элемент списка
* p => new ProductDto { ... } — это лямбда-выражение. p — это один товар из списка, справа — что с ним сделать
* .ToList() — превращаем результат обратно в список
Часть 7. Валидация входных данных
Когда клиент присылает данные через POST или PUT, нужно их проверить. Нельзя доверять тому, что приходит снаружи.
ASP.NET позволяет описать правила прямо в классе модели через атрибуты:
public class Product
{
    public int Id { get; set; }


    [Required(ErrorMessage = "Название обязательно")]
    [MaxLength(255, ErrorMessage = "Название не может быть длиннее 255 символов")]
    public string Name { get; set; }


    [Required]
    [Range(0.01, 1000000, ErrorMessage = "Цена должна быть от 0.01 до 1 000 000")]
    public decimal Price { get; set; }


    [Range(0, int.MaxValue, ErrorMessage = "Количество не может быть отрицательным")]
    public int Stock { get; set; }
}
* [Required] — поле обязательно, не может быть пустым
* [MaxLength(255)] — максимальная длина строки
* [Range(0.01, 1000000)] — значение должно быть в указанном диапазоне
Если [ApiController] стоит над контроллером, валидация происходит автоматически. Если данные не прошли проверку — ASP.NET сам вернёт 400 Bad Request с описанием ошибок, не доходя до твоего кода.
Часть 8. Типичные ошибки новичков
1. Нет await перед асинхронным методом
// НЕПРАВИЛЬНО: возвращает Task, а не список
var products = _db.Products.ToListAsync();


// ПРАВИЛЬНО
var products = await _db.Products.ToListAsync();
Без await переменная будет содержать объект Task (обещание), а не сами данные.
2. SaveChangesAsync не вызван
// НЕПРАВИЛЬНО: товар добавлен только в память EF, но не в базу
_db.Products.Add(product);
return Ok(product);


// ПРАВИЛЬНО
_db.Products.Add(product);
await _db.SaveChangesAsync();
return Ok(product);
EF накапливает изменения в памяти. В базу они попадают только после SaveChangesAsync().
3. Возврат модели БД вместо DTO с чувствительными данными
// ОПАСНО: клиент получит хеш пароля и другие внутренние поля
var user = await _db.Users.FindAsync(id);
return Ok(user);


// ПРАВИЛЬНО: возвращаем только безопасные поля
return Ok(new UserDto { Id = user.Id, Name = user.Name, Email = user.Email });
4. Отсутствие проверки на null после FindAsync
// НЕПРАВИЛЬНО: если товара нет — упадёт с NullReferenceException
var product = await _db.Products.FindAsync(id);
return Ok(product);


// ПРАВИЛЬНО
var product = await _db.Products.FindAsync(id);
if (product == null)
    return NotFound();
return Ok(product);
5. Хранение строки подключения прямо в коде
// НИКОГДА ТАК НЕ ДЕЛАЙ: строка подключения видна всем кто читает код
options.UseSqlServer("Server=prod-server;Password=SuperSecret123;...");


// ПРАВИЛЬНО: читаем из конфигурации
options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection"));
Строки подключения, пароли и секретные ключи никогда не хранятся в коде. Только в конфигурационных файлах, которые не попадают в систему контроля версий (Git).
6. Слишком широкий маршрут возвращает лишние данные
// МЕДЛЕННО и ОПАСНО: загружает всю таблицу в память
var products = await _db.Products.ToListAsync();


// ПРАВИЛЬНО: фильтруй на стороне базы данных
var products = await _db.Products
    .Where(p => p.Stock > 0)
    .OrderBy(p => p.Name)
    .Take(50)
    .ToListAsync();
EF позволяет строить запрос цепочкой методов — это называется LINQ. Каждый метод добавляет условие к SQL-запросу. Запрос выполняется только в момент вызова ToListAsync().

Итог
ASP.NET API — это посредник между фронтендом и базой данных. Он принимает HTTP-запросы, применяет бизнес-логику, работает с базой через Entity Framework и возвращает JSON-ответы.
Ключевые идеи этого раздела:
* HTTP — протокол общения. Запрос содержит метод (GET/POST/PUT/DELETE), путь и тело. Ответ содержит статус-код и тело
* Контроллер — класс с методами, каждый из которых обрабатывает один маршрут
* Entity Framework позволяет работать с базой данных через C#-объекты, без написания SQL вручную
* DbContext — класс-мост между кодом и базой данных
* SaveChangesAsync() — только после этого вызова изменения реально попадают в базу
* DTO — отдельные классы для передачи данных клиенту, чтобы не раскрывать внутреннюю структуру






Frontend
Блок 4. Frontend — HTML + JavaScript
Что такое frontend
Frontend — это всё, что пользователь видит на экране. Кнопки, таблицы, формы, текст — это всё frontend. Он работает в браузере, на компьютере пользователя, а не на сервере.
Браузер понимает три языка:
Язык
	Отвечает за
	HTML
	Структуру страницы — что есть на странице
	CSS
	Внешний вид — как это выглядит
	JavaScript
	Поведение — что происходит когда пользователь что-то делает
	В этом разделе нас интересует главным образом JavaScript — именно он отправляет запросы к API и обновляет страницу когда приходит ответ.
Часть 1. Минимум HTML для понимания
HTML — это язык разметки. Страница описывается тегами. Каждый тег — это элемент страницы.
<!DOCTYPE html>
<html>
  <head>
    <title>Мой магазин</title>
  </head>
  <body>


    <h1>Список товаров</h1>


    <button id="loadBtn">Загрузить товары</button>


    <div id="productList">
      <!-- Сюда JavaScript будет вставлять товары -->
    </div>


    <script src="app.js"></script>


  </body>
</html>
Разберём построчно:
* <!DOCTYPE html> — говорит браузеру что это HTML5-документ
* <html> — корневой элемент, всё внутри него
* <head> — служебная часть, пользователь её не видит. Здесь заголовок страницы, подключение стилей
* <body> — всё что видит пользователь
* <h1> — заголовок первого уровня
* <button id="loadBtn"> — кнопка с идентификатором loadBtn. По этому id JavaScript найдёт её
* <div id="productList"> — пустой блок с идентификатором. JavaScript будет вставлять сюда HTML с товарами
* <script src="app.js"> — подключаем файл с JavaScript-кодом. Всегда ставь в конце body — чтобы к моменту запуска скрипта все элементы уже существовали на странице
Атрибут id
id — это уникальный идентификатор элемента. Он нужен чтобы JavaScript мог найти конкретный элемент на странице. На одной странице не должно быть двух элементов с одинаковым id.
Часть 2. Минимум JavaScript
JavaScript (JS) — язык программирования, который работает прямо в браузере. Он может находить элементы на странице, изменять их содержимое, отправлять запросы к серверу и реагировать на действия пользователя.
Переменные
// let — переменная, значение можно изменить
let count = 0;
count = 5; // можно переназначить


// const — константа, значение нельзя изменить
const apiUrl = 'https://myapp.com/api';
// apiUrl = 'другой адрес'; // ошибка!


// Типы данных
let number = 42;           // число
let text = 'Ноутбук';      // строка
let flag = true;           // булево (true/false)
let nothing = null;        // намеренно пустое значение
Функции
// Обычная функция
function greet(name) {
    return 'Привет, ' + name + '!';
}


let message = greet('Анна'); // message = 'Привет, Анна!'


// Стрелочная функция — более короткая запись
const greet = (name) => {
    return 'Привет, ' + name + '!';
};


// Если функция возвращает одно выражение — можно ещё короче
const greet = (name) => 'Привет, ' + name + '!';
Стрелочные функции => и обычные function делают одно и то же. В современном JS чаще используют стрелочные.
Массивы
// Массив — список значений
const products = ['Ноутбук', 'Мышь', 'Клавиатура'];


// Получить элемент по индексу (нумерация с 0)
console.log(products[0]); // 'Ноутбук'
console.log(products[1]); // 'Мышь'


// Количество элементов
console.log(products.length); // 3


// Перебрать все элементы
products.forEach((product) => {
    console.log(product);
});
Объекты
// Объект — набор свойств
const product = {
    id: 1,
    name: 'Ноутбук',
    price: 75000
};


// Получить свойство
console.log(product.name);    // 'Ноутбук'
console.log(product['name']); // то же самое, другая запись


// Изменить свойство
product.price = 70000;
Массив объектов
Именно так выглядят данные которые приходят из API — массив объектов:
const products = [
    { id: 1, name: 'Ноутбук', price: 75000 },
    { id: 2, name: 'Мышь',    price: 1200  },
    { id: 3, name: 'Монитор', price: 25000 }
];


// Перебрать и вывести имя каждого товара
products.forEach((product) => {
    console.log(product.name);
});
Работа с DOM — поиск элементов на странице
DOM (Document Object Model) — это представление HTML-страницы в виде объектов JavaScript. Через DOM можно находить элементы и менять их содержимое.
// Найти элемент по id
const button = document.getElementById('loadBtn');
const list = document.getElementById('productList');


// Изменить текст внутри элемента
list.textContent = 'Загрузка...';


// Вставить HTML внутрь элемента
list.innerHTML = '<p>Товары загружены</p>';


// Добавить обработчик клика на кнопку
button.addEventListener('click', () => {
    console.log('Кнопка нажата!');
});
Разберём addEventListener:
* Первый аргумент — тип события ('click', 'submit', 'change' и т.д.)
* Второй аргумент — функция, которая запустится когда событие произойдёт
* Эта функция называется обработчик события или callback
Template literals — удобный способ строить строки
const name = 'Ноутбук';
const price = 75000;


// Старый способ — неудобно
const html = '<div>' + name + ' — ' + price + ' руб.</div>';


// Новый способ — template literals (обратные кавычки)
const html = `<div>${name} — ${price} руб.</div>`;
Внутри обратных кавычек ` можно вставлять переменные через ${}. Гораздо читабельнее.
async / await в JavaScript
Так же как в C#, запросы к серверу занимают время. JavaScript не должен замирать пока ждёт ответ. Для этого используют async/await.
// async перед функцией — функция становится асинхронной
const loadProducts = async () => {
    // await — подожди ответа, но не блокируй браузер
    const response = await fetch('https://myapp.com/api/products');
    const products = await response.json();
    console.log(products);
};
Правило то же что в C#: если внутри функции используется await — функция должна быть async.
Часть 3. Fetch API — отправляем запросы к API
fetch — встроенная функция JavaScript для отправки HTTP-запросов. Она работает во всех современных браузерах, ничего устанавливать не нужно.
GET-запрос — получить данные
const loadProducts = async () => {
    const response = await fetch('https://myapp.com/api/products');
    const products = await response.json();
    console.log(products);
};
Разберём построчно:
* fetch('https://...') — отправляет GET-запрос по указанному адресу и возвращает объект response
* response — это не сами данные, а ответ сервера целиком (статус, заголовки, тело)
* response.json() — читает тело ответа и преобразует JSON-текст в JavaScript-объект или массив
* Оба шага асинхронные, поэтому перед обоими стоит await
________________
Проверка статус-кода
Даже если сервер вернул ошибку (например 404 или 500), fetch не выбрасывает исключение — он считает это "успешным" ответом. Нужно проверять вручную:
const loadProducts = async () => {
    const response = await fetch('https://myapp.com/api/products');


    if (!response.ok) {
        // response.ok = true если статус 200-299
        console.error('Ошибка:', response.status);
        return;
    }


    const products = await response.json();
    console.log(products);
};
* response.ok — true если статус-код от 200 до 299, иначе false
* response.status — числовой статус-код (200, 404, 500 и т.д.)
POST-запрос — отправить данные
const createProduct = async (name, price) => {
    const response = await fetch('https://myapp.com/api/products', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json'
        },
        body: JSON.stringify({
            name: name,
            price: price
        })
    });


    if (!response.ok) {
        console.error('Не удалось создать товар');
        return;
    }


    const created = await response.json();
    console.log('Создан товар:', created);
};
Разберём второй аргумент fetch — объект с настройками:
* method: 'POST' — указываем метод запроса. По умолчанию fetch всегда делает GET
* headers — заголовки запроса. 'Content-Type': 'application/json' говорит серверу что тело запроса — это JSON
* body — тело запроса, то что отправляем серверу
* JSON.stringify({...}) — преобразует JavaScript-объект в JSON-строку. Это обратная операция к response.json()
PUT-запрос — обновить данные
const updateProduct = async (id, name, price) => {
    const response = await fetch(`https://myapp.com/api/products/${id}`, {
        method: 'PUT',
        headers: {
            'Content-Type': 'application/json'
        },
        body: JSON.stringify({ name, price })
    });


    if (!response.ok) {
        console.error('Не удалось обновить товар');
        return;
    }


    const updated = await response.json();
    console.log('Обновлён:', updated);
};
* `https://.../api/products/${id}` — подставляем id прямо в URL через template literal
* { name, price } — сокращённая запись. Когда имя переменной совпадает с именем свойства, можно писать просто { name, price } вместо { name: name, price: price }
DELETE-запрос — удалить данные
const deleteProduct = async (id) => {
    const response = await fetch(`https://myapp.com/api/products/${id}`, {
        method: 'DELETE'
    });


    if (!response.ok) {
        console.error('Не удалось удалить товар');
        return;
    }


    console.log('Товар удалён');
    // Ответ пустой (204 No Content), json() не вызываем
};
При DELETE сервер возвращает 204 No Content — тела нет, вызывать response.json() не нужно.
Часть 4. Собираем всё вместе — полный пример
Теперь соберём реальное мини-приложение: загружаем список товаров из API и показываем на странице. Плюс форма для добавления нового товара.
HTML-файл (index.html)
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <title>Список товаров</title>
    <style>
        body { font-family: sans-serif; max-width: 600px; margin: 40px auto; }
        .product { border: 1px solid #ddd; padding: 12px; margin-bottom: 8px; border-radius: 4px; }
        .product h3 { margin: 0 0 4px; }
        button { padding: 8px 16px; cursor: pointer; }
        input { padding: 8px; margin-right: 8px; width: 200px; }
        #error { color: red; }
    </style>
</head>
<body>


    <h1>Товары</h1>


    <button id="loadBtn">Загрузить товары</button>


    <div id="error"></div>
    <div id="productList"></div>


    <hr>


    <h2>Добавить товар</h2>
    <input id="nameInput"  type="text"   placeholder="Название" />
    <input id="priceInput" type="number" placeholder="Цена" />
    <button id="addBtn">Добавить</button>


    <script src="app.js"></script>


</body>
</html>
JavaScript-файл (app.js)
// Адрес нашего API — меняем только здесь если адрес изменится
const API_URL = 'https://myapp.com/api/products';


// Находим все нужные элементы на странице
const loadBtn    = document.getElementById('loadBtn');
const addBtn     = document.getElementById('addBtn');
const productList = document.getElementById('productList');
const nameInput  = document.getElementById('nameInput');
const priceInput = document.getElementById('priceInput');
const errorDiv   = document.getElementById('error');


// ---- Функции для работы с API ----


// Загрузить все товары
const loadProducts = async () => {
    errorDiv.textContent = '';
    productList.textContent = 'Загрузка...';


    const response = await fetch(API_URL);


    if (!response.ok) {
        errorDiv.textContent = 'Не удалось загрузить товары. Статус: ' + response.status;
        productList.textContent = '';
        return;
    }


    const products = await response.json();
    renderProducts(products);
};


// Создать новый товар
const createProduct = async (name, price) => {
    const response = await fetch(API_URL, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ name, price: Number(price) })
    });


    if (!response.ok) {
        errorDiv.textContent = 'Не удалось добавить товар. Статус: ' + response.status;
        return;
    }


    // После успешного создания — перезагружаем список
    nameInput.value  = '';
    priceInput.value = '';
    await loadProducts();
};


// Удалить товар по id
const deleteProduct = async (id) => {
    const response = await fetch(`${API_URL}/${id}`, {
        method: 'DELETE'
    });


    if (!response.ok) {
        errorDiv.textContent = 'Не удалось удалить товар';
        return;
    }


    // После удаления — перезагружаем список
    await loadProducts();
};


// ---- Функция для отрисовки товаров ----


const renderProducts = (products) => {
    // Если товаров нет — показываем сообщение
    if (products.length === 0) {
        productList.textContent = 'Товаров пока нет';
        return;
    }


    // Строим HTML для каждого товара и склеиваем в одну строку
    const html = products.map((product) => {
        return `
            <div class="product">
                <h3>${product.name}</h3>
                <p>Цена: ${product.price} руб.</p>
                <button onclick="deleteProduct(${product.id})">Удалить</button>
            </div>
        `;
    }).join('');


    // Вставляем весь HTML в контейнер
    productList.innerHTML = html;
};


// ---- Обработчики событий ----


// Клик на кнопку "Загрузить товары"
loadBtn.addEventListener('click', () => {
    loadProducts();
});


// Клик на кнопку "Добавить"
addBtn.addEventListener('click', () => {
    const name  = nameInput.value.trim();
    const price = priceInput.value.trim();


    // Простая проверка на стороне клиента
    if (!name || !price) {
        errorDiv.textContent = 'Заполните название и цену';
        return;
    }


    errorDiv.textContent = '';
    createProduct(name, price);
});
Разберём важные моменты в этом коде:
products.map() — метод массива который трансформирует каждый элемент. В отличие от forEach он возвращает новый массив. Здесь каждый объект-товар превращается в HTML-строку.
.join('') — склеивает массив строк в одну строку без разделителей. Получается один большой HTML-блок.
productList.innerHTML = html — вставляем весь HTML в контейнер. Браузер разберёт строку и создаст реальные DOM-элементы.
onclick="deleteProduct(${product.id})" — прямо в HTML-строке вставляем вызов функции с нужным id. Когда пользователь нажмёт кнопку, браузер вызовет deleteProduct(1) (или с любым другим id).
.trim() — убирает пробелы в начале и конце строки. Нужно чтобы строка из одних пробелов не прошла проверку if (!name).
Number(price) — преобразует строку в число. Из input всегда приходит строка, даже если тип поля number. API ожидает число, поэтому явно конвертируем.

Часть 5. Как фронтенд и бэкенд живут отдельно
Важно понять: фронтенд и бэкенд — это два разных приложения, которые работают независимо.
Фронтенд-файлы (HTML, CSS, JS) могут лежать где угодно — на отдельном сервере, в CDN, просто на диске. Они приходят к пользователю один раз при открытии страницы. Дальше JavaScript в браузере сам ходит к API.
CORS — почему запросы иногда блокируются
Браузер по соображениям безопасности не позволяет JavaScript с одного адреса обращаться к серверу на другом адресе. Это называется CORS (Cross-Origin Resource Sharing).
Например, фронтенд работает на http://localhost:3000, а API — на http://localhost:5000. Это разные адреса, браузер заблокирует запрос.
Чтобы это исправить, нужно на стороне ASP.NET API разрешить запросы с адреса фронтенда. В Program.cs:
// Разрешить запросы от фронтенда
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.WithOrigins("http://localhost:3000") // адрес фронтенда
              .AllowAnyHeader()
              .AllowAnyMethod();
    });
});


var app = builder.Build();


app.UseCors("AllowFrontend"); // применить политику
Разберём построчно:
* AddCors(...) — регистрируем политику CORS в приложении
* "AllowFrontend" — имя политики, придумываем сами
* WithOrigins("http://localhost:3000") — разрешаем запросы именно с этого адреса
* AllowAnyHeader() — разрешаем любые заголовки запроса
* AllowAnyMethod() — разрешаем любые методы (GET, POST, PUT, DELETE)
* app.UseCors("AllowFrontend") — подключаем политику к конвейеру обработки запросов
Часть 6. Типичные ошибки новичков
1. Забыли await
// НЕПРАВИЛЬНО: products будет объектом Promise, а не массивом
const products = fetch(API_URL).json();


// ПРАВИЛЬНО
const response = await fetch(API_URL);
const products = await response.json();
2. Вызвали response.json() дважды
// НЕПРАВИЛЬНО: тело ответа можно прочитать только один раз
const response = await fetch(API_URL);
console.log(await response.json()); // первый раз — ОК
console.log(await response.json()); // второй раз — ошибка, поток уже закрыт


// ПРАВИЛЬНО: сохраняем в переменную
const response = await fetch(API_URL);
const data = await response.json();
console.log(data);
3. Не указали Content-Type при POST/PUT
// НЕПРАВИЛЬНО: ASP.NET не поймёт что пришёл JSON и вернёт 415 Unsupported Media Type
const response = await fetch(API_URL, {
    method: 'POST',
    body: JSON.stringify({ name: 'Ноутбук', price: 75000 })
});


// ПРАВИЛЬНО
const response = await fetch(API_URL, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ name: 'Ноутбук', price: 75000 })
});
4. Забыли JSON.stringify при отправке
// НЕПРАВИЛЬНО: отправится строка "[object Object]", а не JSON
body: { name: 'Ноутбук', price: 75000 }


// ПРАВИЛЬНО
body: JSON.stringify({ name: 'Ноутбук', price: 75000 })
5. Вставляют данные из API напрямую в innerHTML без очистки
// ОПАСНО: если name содержит <script>...</script>, он выполнится
productList.innerHTML = `<div>${product.name}</div>`;


// БЕЗОПАСНЕЕ: использовать textContent для текстовых данных
const div = document.createElement('div');
const h3 = document.createElement('h3');
h3.textContent = product.name; // textContent не интерпретирует HTML
div.appendChild(h3);
productList.appendChild(div);
Вставка данных из внешнего источника через innerHTML — потенциальная уязвимость XSS (Cross-Site Scripting). Для учебного проекта это не критично, но важно знать о проблеме.
6. Не обрабатывают ошибки сети
// НЕПРАВИЛЬНО: если сервер недоступен — страница просто зависнет
const response = await fetch(API_URL);


// ПРАВИЛЬНО: оборачиваем в try/catch
try {
    const response = await fetch(API_URL);


    if (!response.ok) {
        throw new Error('Ошибка сервера: ' + response.status);
    }


    const products = await response.json();
    renderProducts(products);


} catch (error) {
    // Сюда попадём если сервер недоступен или response.ok = false
    errorDiv.textContent = 'Произошла ошибка: ' + error.message;
}
try { ... } — пробуем выполнить код
catch (error) { ... } — если что-то пошло не так — попадаем сюда

Итог
Frontend — это то, что живёт в браузере пользователя. Он не хранит данные и не содержит бизнес-логику. Его задача — показать интерфейс и общаться с API.
Ключевые идеи этого раздела:
* HTML описывает структуру страницы, JavaScript управляет её поведением
* document.getElementById находит элемент на странице по его id
* fetch отправляет HTTP-запросы к API — GET, POST, PUT, DELETE
* Ответ приходит в два шага: сначала await fetch(...), потом await response.json()
* При POST и PUT обязательно указывать Content-Type: application/json и оборачивать данные в JSON.stringify
* CORS нужно настраивать на стороне ASP.NET если фронтенд и API живут на разных адресах
* Всегда проверяй response.ok и оборачивай запросы в try/catch
